Introduction

iGrOw is a program packed in a html-document that calculates the groundwater flow to an open water drain in a vertical cross section.

The philosophy behind is to be as interactive as possible. Parameters are set through menus. To make this feasible, for all settings a minimal number of parameters is chosen. For most menu entries, a change of parameter results in an immediate adaption on the screen.

As output the program produces in the browser window three views (positioned vertically underneath each other, so the vertical scroll bar has to be used to go from one to the other), illustrated by in the picture below:

  • a global view: showing the total system from the water divide on the left to the water divide

  • a local view: showing the system near the open water on the left

  • a number view: interesting input and output numbers are combined in a table.

Moreover iGrOw uses seven graphical menus as interface.

The model bottom

The bottom of an iGrOw model can either be:

  • closed, i.e. impermeable
  • open: allowing fluxes through the bottom connecting the iGrOW model with a (non depicted) layer below. The parameters determining this connection are given in the hydro-vars-menu and discussed there in this section.

Catchment form

iGrOw models the flow in a vertical plane orthogonal to the drain. This option makes assumptions on the direction orthogonal to that plane.

  • the option parallel assumes that what iGrOw calculates is (in first order) representative for straight lines orthogonal to the iGrOw plane
  • the options divergent and convergent assume that what iGrOw calculates is (in first order) representative for circles orthogonal to the iGrOw plane, respectively with center in the middle of the drain, i.e. at the left end of the domain or at the right end of the domain . The parameter determining the non-parallel choices is called catchment curvature and is discussed in the geomenu.

Entering parameters

An iGrOw model is dertermined by (rather small) number of parameters. The user can choose values for these parameters using the menus, which will be discussed in the following sections.

In some browsers, values in these menus can be changed using small arrows at the right of the parameter field or using the up and down arrow keys. Using these techniques will change the value of the parameter gradually.

In many case the user may want to just to enter a specific number for the parameter. Suppose for instance the user wants the change the width of the iGrOw domain from 120 to 105. The most standard way of doing this is by looking upon the cell in the menu as a "mini"-editor: first removing the 0 and 2 and then entering the 0 and 5. In doing so the content of the cell becomes 12, 1, 10 and finally 105. It is certainly not the intention of the user to make iGrOW accept these intermediate values 12, 1 and 10 and process them by updating the screen and/or other calculations. For that reason iGrOw only accept the current cell value only as final if the enter/return key is hit. Intermediate non-final values are signaled to the user by a green color.



A similar approach to oppose committing values until the enter/return key is hit is used for several values grouped in order to let the group satisfy relation between parameters, as will be discussed in a following section.

This delay in passing on parameters from the menu to iGrOw is not needed for the numerical parameters, as by construction they are only passed on collectively if the "make discretization" button is hit.

The dimensions of the model

The following figure clarifies the meaning of the four first values in this menu who determine the dimensions of the model:



These values are:
  • domain width (L/2): the width of vertical model section; the (L/2) is added to remind the user of the classical view that on the right hand side of the water divide one can think of a mirror image of the one modeled here in the window (with a drain on the far right), so that in that way the "whole" model has a width of L.
  • drain width (B/2): the width is measured to the top of the drain, so it measures the "dry" width; the (B/2) suggests a mirror drain to the right of the left water divide
  • domain height: measured with respect to the bottom of the model, which is assumed to have height 0
  • drain bottom: height of the bottom of the drain with respect of the bottom of the model.

Changing a value in this geo-menu has an immediate effect on all plots in the window as can be seen in the following picture:

All these values have preset minimum and maximum values (set in the script "iGrOwmenu.js"). There are also relations between the settings that need to be satisfied. For instance, the drain bottom should never be higher than the domain height. The user is warned if he nevertheless tries to do this. So if one wants to lower the domain height and the drain bottom in such a way that the new domain height is below the current drain bottom, one should first lower the drain bottom to avoid this type of conflicts.

Catchment curvature

For a parallel the width of the catchment WWW draining is considered to be constant W(x)=W(0)W(x) = W(0)W(x)=W(0).

If one chooses a convergent (towards the drain) or a divergent (towards to the drain), this width is changing linearly with the distance to the drain:

  • convergent: W(x)=(1+x)W(0)W(x) = (1+\kappa\; x)\; W(0)W(x)=(1+x)W(0)
  • divergent: W(x)=(1x)W(0)\;\;\,W(x)= (1-\kappa\; x)\;W(0)W(x)=(1x)W(0)

The parameter \kappa is called the curvature of the catchment.

In the divergent case, there is the condition on the curvature:

Constraints on the parameters

Non feasible parameter values

Not all values of parameters result in feasible models. For instance, the drain level should always be above the drain bottom. The menu's however allow to change parameter values into non-feasible models. iGrOw informs the user of this non-feasibility and refuses to continue until all parameters are feasible.

The following image shows such a setting: a drain bottom of 8.1 is increased to 11 (and the enter/return button was hit passing this number on to iGrOw) This puts the bottom at the domain height and above the drain water level, both making it impossible for iGrOw to continue. iGrOw signals this to the user by a window, allowing the user the choice to accept this change or not. If accepting the set values, the parameters involved in the non-feasibility are colored red in the menu's. After changing the domain height into 12 and the drain level into 11.4, the parameter setting is feasible for iGrOW and the red warning colors disappear.

A better way of changing many values in iGrOw (avoiding error messages) is to postpone hitting the enter/return key until all the relevant parameters have received they proper value, as in the following figure (see also section):

A non-feasible parameter value can also be the result of a typing error, as in the following figure where the drain bottom is accidentally increased to 118.1. In that case the user can not accept this setting. iGrOw reacts to this choice by allowing the user to reset the parameter values to the last feasible ones.

Parameter constraints in iGrOw

In order to be feasible, the parameters should satisfy the following constraints:

  • geo.drainwidth <= 0.9 *geo.domainwidth
    the 0.05 is needed as iGrOw wants to put nodes to the right of the drain
  • solution.CELLwidth >= geo.drainwidth
  • geo.drainbottom > 0.1 * geo.domainheight;
    the 0.05 is needed as iGrOw wants to put nodes underneath drain bottom
  • physics.drainlevel >= geo.drainbottom+ 0.05
    iGrOw assumes a minimal water level of 0.05 in the drain
  • physics.drainlevel <= geo.domainheight -0.05
  • geo.drainwidth - geo.traptalud * (geo.domainheight - geo.drainbottom) >= 0
    see the section on trapezoidal form
  • geo.curvature < geo.domainwidth
    only relevant if for the catchment form convergentwas chosen
  • the conductivity matrix [kxxkxzkxzkzz] \begin{bmatrix} k_{xx} & k_{xz}\\ k_{xz}& k_{zz} \end{bmatrix} [kxxkxzkxzkzz] (see also the section on the equations) should only have positive eigenvalues
  • physics.cseepage <= physics.cdrain
    only relevant if the option height interpolated and/or the option seepage face was chosen

The flow equations




The problem to solve is to calculate for each point (x,z)(x,z)(x,z) below the groundwater surface the groundwater head H(x,z)H(x,z)H(x,z). In the figure above the variation of HHH is indicated by colors.

Calculating the position of the groundwater surface is part of the problem to be solved. The groundwater surface consists of the points (x,z)(x,z)(x,z) for which the following equation holds:

H(x,z)=zH(x,z) = z H(x,z)=z

The flux is calculated by:

Q=[QxQz]=[kxxkxzkxzk_zz][H/xH/z]W(x)\vec{Q}= \begin{bmatrix} Q_x \\ Q_z \end{bmatrix} = - \begin{bmatrix} k_{xx} & k_{xz} \\ k_{xz} & k\_{zz} \end{bmatrix} \begin{bmatrix} \partial H/\partial x\\ \partial H/\partial z \end{bmatrix}\; W(x) Q=[QxQz]=[kxxkxzkxzk_zz][H/xH/z]W(x)

were W(x)W(x)W(x) is the width of the catchment as discussed in the catchment curvature section.

The mass balance equation is in each point (x,z)(x,z)(x,z) given by:

Qxx(x,z)+Qzz(x,z)=q(x,z) \frac{\partial Q_x}{\partial x}(x,z) + \frac{\partial Q_z}{\partial z}(x,z)={q}(x,z) xQx(x,z)+zQz(x,z)=q(x,z)

The external flux has different expressions on different parts of the domain:

  1. in a point AAA in the interior of the domain no external flux is present:

    q(A)=0q(A) = 0 q(A)=0

  2. in a point BBB on the groundwater surface, the external flux is given by the recharge intensity rrr:

    q(B)=rW(xB)q(B) = r\; W(x_B) q(B)=rW(xB)

  3. the left and right boundaries are considered as water divides, so no water enters or leaves the system there so that for a point CCC:

    q(C)=0q(C) = 0 q(C)=0

  4. At a point DDD on the lower boundary, two choices are possible:

  • when model bottom is closed, the flux is just zero:

    q(D)=0q(D) = 0 q(D)=0

  • is the model bottom is chosen to be open the system is contact with a lower groundwater layer called base layer with a constant head HbaseH_{base}Hbase. For any point DDD on this lower boundary the flux is then given by:

    q(D)=HbaseH(D)cbaseW(Dx)q(D) = \frac{H_{base}-H(D)}{c_{base}}\; W(D_x) q(D)=cbaseHbaseH(D)W(Dx)

    where cbasec_{base}cbase is the resistance of the exchange

  1. at points as EEE where the system is contact with the water in the drain, so below the water level in the drain, (with constant head HdrainH_{drain}Hdrain = drain_level, the flux is calculated by:

    q(E)=HdrainH(E)c(zE)W(Ex)q(E) = \frac{H_{drain}-H(E)}{c(z_E)}\; W(E_x) q(E)=c(zE)HdrainH(E)W(Ex)

    for c(zE)c(z_E)c(zE) see the section on outflow resistance

  2. at points as FFF on the wall of the drain above the water level, the flux is given by:

  • if the option seepage face is chosen:

    q(F)=zFH(F)cseepageW(Fx)q(F) = \frac{z_F-H(F)}{c_{seepage}}\; W(F_x) q(F)=cseepagezFH(F)W(Fx)

  • if the option seepage face) is not selected:

    q(F)=0 q(F) = 0 q(F)=0

  1. if the option dirichlet is chosen, a fixed head boundary condition is chosen:

H(Gx,Gz)=Gz H(G_x,G_z) = G_z H(Gx,Gz)=Gz

and no balance equation is solved

The numerical settings

The numeric menu

As the equations can not be solved analytically, a numerical solution will be calculated. The first step in this procedure is make a discretization of space. A finite number of nodes will be placed in the domain, and only in these nodes an approximation to the HHH's will be calculated. This (initial) discretization is done in the numerical menu with the help of the parameters in that menu, as illustrated in the following figure:

The role of these parameters is illustrated in the following figure:


As the solution will show dramatically different variations throughout the solution domain, with very horizontal flow on the right and with strongly curved flow near the bottom of the drain, the number of nodes left and right can be independently chosen.

A critical point is the rightmost point of the drain. A # dens factor greater then 0 (maximum value is 1) makes the distribution of the nodes around that point denser.

The nodes shown in the figures above are the initial nodes. During the solution procedure they will shift position in order to capture the positions with groundwater present.

Discretization cells

The approximation of the heads H(x,z)H(x,z)H(x,z) will be calculated at and only at the discretization nodes. These heads are the solution of balances constructed at these nodes by a classical Finite Element technique.

Finite Element technique requires the nodes to be connected by triangles. iGrOw uses the classical Delauney technique for this. Hitting the make discretization button also calls this method for the constructed nodes. The result of this can be seen by checking the show cells button in the numerical menu:


One usually tries to make the triangles for Finite Elements as equilateral as possible. For the problems iGrOW is tackling, this given the x-z ratio not possible. To appreciate a generated triangularization, one should not forget to take also a view in the physical aspect ratio as in the following picture:

Calculation of the numerical solution and adaptation of the nodes

Once a discretization made, a numerical solution can be calculated. As the differential equations (see above) are essentially balance equations, the solving procedure tries iteratively to find head values in the nodes (HiH_iHi) such that the numerical mass balances for each node the numerical mass balance error is minimized. The node inspector option of the solution menu gives a (local) view upon results of such a balance.


But iGrOw not only has find the location of the ground water table. This requires a second iteration, during which the position of the nodes (and cells) is adapted to better fit this ground water table. If the option seepage face is chosen, nodes also move along the wall of the drain in order to optimal locate this seepage face. The figure also shows that in iGrOw nodes can never rise above the domain height.

Fluxes

Checking the draw fluxes option draw fluxes both in the top global and middle local view. The user has to select a factor to map the flux vectors (who have a different physical dimension: length/time as the pure length dimensions of the axes) onto the plot. The right border of this cell is plotted as a yellow line. The results of the analysis are shown in a small report (that can be dragged to any place in the window) as in the following figure:

This report contains the following information:

  • Cell H_mean: the mean value of the top nodes in the cell

Running cases/uploading from CSV

The manual discussed until now how models could be made interactively (through menus) and stored in tables. To run many cases, the interactive approach may be too cumbersome.

For that reasons, iGrOw allows also to define all these cases in a csv file and upload it in iGrOw. iGrOw will run all these cases and store them in a table.

For this procedure, one should first make a csv file with all input variables as column names and where the rows correspond to the cases. Then hitting the -"Upload CSV to table"_ and selecting an input file runs all the examples of the chosen csv:

To help the user in this, the folder "CSV" contained in the install contains some examples:

  • iGrOwInputExample.csv: contains an input csv file with one run defined
  • iGrOwMultiInputExample.csv: contains an input csv with several lines as input. The folder contains also the corresponding iGrOwMultiOutput.csv, generated with the Download table to CSV(#downloadtabletocsv) after all models were calculated.
  • a template file that may be used as a starter for making input csv files.

As the spelling of the columns names is VERY IMPORTANT the use of this template file is advised.

There maybe fields that are not relevant. For instance if one choses a smooth crosssection, then there is no need for a value for the trap_talud field. It is advised to fill the field then with "NaN" (not a number).

As special case in these input csv files is the _CELL width" field. If it is NaN no cell analysis will be performed, if it contains the length of the CELL, cell analysis will be performed and presented in the output.

If the parameters do not satisfy the constraints, output parameters are not generated and the comment field will contain the message "not run".

Dump all to json-file

The table contains only some calculated pre-chosen values, and not all values within the model. The last button allows one to download all the values used and calculated values in a file. The format chosen is "json". This is a very general format for which most programming languages have a "reader" for this type of file:

  • R: "fromJSON" in the "rjson" package
  • Python: "json.load" in the "json" module
  • Excel: "Select Data > Get Data > From File > From JSON"
  • ...

Once read into a language proper structure, all further calculations can be done withing the chosen language.

Non-menu settings

The sections above discussed values of variables that can be interactively set through the menu options.

The iGrOw uses on start up values that can be changed if wanted. The are collected in a javascript file (which can be opened and edited by any editor) called iGrOwdefaultsettings.js that can be found the iGrOw-scripts folder. The following image shows the first and the last part of this file as provided in the of the installation:

......

All items but the last one have been discussed and explained above.

The "convergence" structure collects numbers that are used in the mathematics of the solution procedure, and should not normally be changed:

  • omega: classical relaxation factor used in the iterative solution of the equations. Larger numbers can result in a faster solution, but can also result in non-convergence.

  • alpha: relaxation factor used in repositioning the nodes during the solution procedure

  • baserelax: relaxation factor used in the part of the solution that handles the base flow. Larger numbers can result in a faster solution, but can also result in non-convergence.

  • maxiter1: maximum number iterations in the solutions of the main part of the equations

  • maxiter2: maximum number iterations in the solutions of base part of the equations

  • maxHchange: stop criterion for heads: if changes in H-values are smaller than this number, the iteration in the main solution procedure stops

  • maxQchange: stop criterion for fluxes: if changes in Q-values of the base values are smaller than this number, the iteration in the solution procedure that handles the base fluxes stops

  • maxdiffHY: stop criterion for the adaption of the grid.

In principle, any of these numbers in this file can be changed. If an altered file is saved in the script folder, it will be used the next time the iGrOw is (re)opened.
